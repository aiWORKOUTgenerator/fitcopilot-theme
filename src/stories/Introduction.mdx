# FitCopilot Design System

Welcome to the FitCopilot component library and design system, the single source of truth for our UI development. This Storybook serves as the interactive documentation for all design tokens, components, and patterns used in the FitCopilot platform.

## Design Principles

Our design system follows these core principles:

1. **Accessibility First**: Every component meets WCAG 2.1 AA standards, ensuring usability for all users.
2. **Consistency Over Customization**: We prioritize consistent experiences over one-off customizations.
3. **Progressive Enhancement**: Components work with or without JavaScript, enhancing the experience when available.
4. **Performance Matters**: We optimize for speed, keeping components lightweight.
5. **Responsive By Default**: All components adapt seamlessly to any screen size.

## Component Architecture

FitCopilot follows a feature-first approach to component organization:

- **UI Components**: Reusable, presentational elements (buttons, inputs, cards)
- **Layout Components**: Structural components (grid, containers, sections)
- **Feature Components**: Domain-specific components with business logic

## Component Anatomy

Each component follows a consistent structure:

1. **Props Interface**: Clear TypeScript interfaces defining all accepted props
2. **Default Exports**: Components are named exports for better tree-shaking
3. **Styling**: Component styling through Tailwind utility classes
4. **Documentation**: Comprehensive JSDoc comments

```tsx
interface ButtonProps {
  /** The button's label content */
  label: string;
  /** Whether the button has primary styling */
  primary?: boolean;
  /** The button's size */
  size?: 'small' | 'medium' | 'large';
  /** Click handler */
  onClick?: () => void;
}
```

## Navigation

The sidebar is organized to match our component hierarchy:

- **Foundations**: Design tokens, colors, typography, and spacing
- **UI**: Basic UI components like buttons, inputs, and cards
- **Layout**: Structural components like containers, grids, and sections
- **Features**: Complex feature components organized by section (Homepage, Workouts, etc.)

## Contribution Workflow

To add or update component stories:

1. **Audit**: Run `npm run storybook:audit` to identify components missing stories
2. **Generate**: Use `npm run storybook:generate src/path/to/Component.tsx` for new stories
3. **Document**: Fill in component descriptions and prop documentation
4. **Test**: Ensure your story passes a11y tests and works in all viewports
5. **Review**: Submit your changes for design review

## Accessibility Standards

All components must:

- Include proper semantic HTML elements
- Support keyboard navigation
- Maintain color contrast ratios (4.5:1 minimum)
- Include ARIA attributes when appropriate
- Work with screen readers

## Version Control

This design system follows semantic versioning:

- **Major**: Breaking changes to component APIs
- **Minor**: New components or enhanced features
- **Patch**: Bug fixes and non-breaking improvements
